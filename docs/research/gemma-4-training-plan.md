# Gemma 4 Post-Training for AgentShield Sigma Rule Generation

**Date:** 2026-04-06
**Status:** Research / Proposal

---

## Executive Summary

Gemma 4 is a strong candidate for building a fine-tuned, locally-executable Sigma rule generator. The model family — released under **Apache 2.0** — offers frontier-class performance at sizes that run on consumer hardware. The recommended approach is **SFT cold-start + GRPO reinforcement learning** using the 26B MoE variant (only 4B active parameters) as the primary target, with the 31B dense model as a quality ceiling. This creates a defensible moat through the combination of proprietary training data, a verified reward pipeline, and optimized on-device deployment.

### Architecture Highlights

- **Per-Layer Embeddings (PLE):** The E2B/E4B models feed a secondary embedding signal into every decoder layer, giving a 2.3B-active model the representational depth of 5.1B total parameters. Fits in <1.5GB quantized.
- **Hybrid Attention:** Interleaves local sliding-window attention with full global attention (final layer always global). Good for long-context security log analysis.
- **MoE (26B variant):** Only 4B parameters active per token out of 26B total — frontier-level performance at a fraction of the compute. LMArena score: **1441** (comparable to GPT-4 class).

## 2. Training Strategy: SFT + GRPO Reinforcement Learning

### Why Not Pure SFT?

SFT alone teaches format and basic competence but **caps quality at the demonstration ceiling** — the model can only be as good as the examples it's trained on. For Sigma rule generation, we need the model to reason about novel attack patterns and compose correct detection logic, which requires going beyond imitation.

### Why GRPO (Group Relative Policy Optimization)?

GRPO, pioneered by DeepSeek, is the current best practice for RL post-training of open models:

- **No critic model needed** — ~50% memory reduction vs PPO. Critical for training on the 26B MoE.
- **On-policy sampling** — generates fresh completions each iteration, avoiding the distribution shift problems of DPO.
- **Verifiable rewards** — perfect for Sigma rules where we can programmatically check correctness.
- **Proven at scale** — used in DeepSeek-R1, Gemma 3's own post-training, and numerous community projects.

### Recommended Pipeline

```
Phase 1: Cold-Start SFT (Format + Domain Knowledge)
    ↓
Phase 2: GRPO with Verifiable Rewards (Quality + Reasoning)
    ↓
Phase 3: Rejection Sampling → Final SFT Polish
    ↓
Phase 4: Quantize + Deploy
```

#### Phase 1 — Cold-Start SFT

**Goal:** Teach the model Sigma rule syntax, AgentShield conventions, and basic threat-to-rule mapping.

**Data requirements:**
- **1,000–5,000 high-quality examples** of (threat description → Sigma rule) pairs
- Sources:
  - Our existing 45 Sigma rules + manually written explanations
  - SigmaHQ's full rule corpus (~3,000+ rules) with reverse-engineered descriptions
  - Synthetic pairs generated by Claude/GPT-4 from MITRE ATT&CK descriptions
  - Expert-annotated "gold" examples for AI agent-specific threats
- **Format:** Each example should include the threat context, reasoning about what to detect, and the complete YAML rule output

**Compute:** ~8-16 GPU-hours on A100 for the 26B MoE with QLoRA (rank 64-128)

**Framework:** Hugging Face TRL `SFTTrainer` or Unsloth (2x speed optimization)

#### Phase 2 — GRPO Reinforcement Learning

**Goal:** Push rule quality beyond the SFT ceiling. Teach the model to reason about detection logic, minimize false positives, and handle novel attack patterns.

**Reward function design (the core of the moat):**

```python
def sigma_rule_reward(generated_rule: str, prompt: str) -> float:
    score = 0.0

    # 1. Syntactic validity (0.0 - 0.3)
    #    Parse as YAML, check required Sigma fields exist
    if valid_yaml(generated_rule):
        score += 0.1
    if has_required_fields(generated_rule):  # title, id, logsource, detection, condition
        score += 0.1
    if passes_sigma_check(generated_rule):   # sigma check -e
        score += 0.1

    # 2. Semantic correctness (0.0 - 0.4)
    #    Does the detection logic actually match the described threat?
    if condition_references_valid(generated_rule):  # all condition identifiers exist
        score += 0.1
    if logsource_correct(generated_rule):           # product: ai_agent, category: agent_events
        score += 0.1
    if detection_fields_plausible(generated_rule):  # field names exist in schema
        score += 0.1
    if tags_match_threat(generated_rule, prompt):   # MITRE ATT&CK tags align
        score += 0.1

    # 3. Quality signals (0.0 - 0.3)
    #    Format compliance (our conventions), specificity, false-positive consideration
    if follows_naming_convention(generated_rule):   # ai_agent_*.yml pattern
        score += 0.05
    if has_falsepositives_section(generated_rule):
        score += 0.05
    if has_references(generated_rule):
        score += 0.05
    if detection_is_specific(generated_rule):       # not overly broad wildcards
        score += 0.05
    if level_appropriate(generated_rule, prompt):   # severity matches threat
        score += 0.1

    return score
```

**Data requirements:**
- **10,000–50,000 threat description prompts** (no output labels needed — GRPO generates its own)
- Sources:
  - MITRE ATT&CK technique descriptions
  - CVE descriptions mapped to detection scenarios
  - AI agent-specific attack scenarios (prompt injection, tool poisoning, data exfil, etc.)
  - Synthetic threat scenarios generated at varying difficulty levels

**GRPO hyperparameters:**
- Group size: 16 (sample 16 completions per prompt)
- KL penalty: β = 0.04
- Clip ratio: ε = 0.2
- Batch size: 128 prompts per step
- Training: ~2,000–5,000 steps

**Compute:** ~200-500 GPU-hours on A100/H100 for the 26B MoE (LoRA rank 128)

**Framework:** TRL `GRPOTrainer` or OpenRLHF (better for multi-node)

#### Phase 3 — Rejection Sampling + Final SFT

**Goal:** Stabilize the RL-trained model and create a polished final version.

1. Generate 100 rules per prompt across 1,000 diverse prompts
2. Score each with the reward function
3. Keep top-5% (5,000 "gold" rules)
4. Final SFT pass on these gold rules mixed with 20% general instruction data (prevents catastrophic forgetting)

**Compute:** ~16-32 GPU-hours

#### Phase 4 — Quantize + Deploy

See Section 4 below.

## 3. The Moat — What's Actually Defensible

### Weak moats (easily replicated):
- "We fine-tuned a model" — anyone with the data can do this in a weekend
- "We use Gemma 4" — it's open source under Apache 2.0
- The fine-tuning technique itself — GRPO is well-documented

### Strong moats (hard to replicate):

1. **The reward function + evaluation pipeline.** Our existing pytest conformance suite (6 tests validating syntax, required fields, references, condition cross-references, unique IDs, tactic tag format) becomes the backbone of a verifiable reward signal. This is domain expertise encoded as code. Competitors would need to independently develop equivalent validation logic.

2. **The curated training data.** Our 45 expert-written AI agent Sigma rules are the seed crystal. Combined with SigmaHQ's corpus, expert annotations, and iteratively-improved synthetic data (using rejection sampling from our own model), we build a proprietary dataset that improves with each training cycle.

3. **The continuous improvement flywheel:**
   ```
   Users generate rules → Experts validate → Validated rules become training data
        → Model improves → Better rules generated → Repeat
   ```

4. **On-device deployment optimization.** A model specifically quantized, profiled, and tested for M4 Max / RTX 4090 / RTX 5090 hardware — with measured latency, quality metrics at each quantization level, and deployment packaging — represents significant engineering effort.

5. **Domain-specific evaluation benchmarks.** Creating a "SigmaEval" benchmark — a held-out test set of threat descriptions with known-correct Sigma rules, graded on syntax validity, detection accuracy, false-positive rate, and convention compliance — is itself a moat. You can't improve what you can't measure.

### RAG vs Fine-tuning consideration:

RAG (stuffing rules into context) can approximate fine-tuning for simple cases, but:
- Fine-tuning encodes **generative patterns** (how to compose detection logic), not just retrieval
- RL teaches **reasoning about novel attacks** that aren't in any context window
- On-device deployment favors small, self-contained models over large context windows + retrieval infrastructure
- For the local execution story, fine-tuning is the clear winner

### Target Hardware Profiles

| Profile | Hardware | Budget | Use Case |
|---------|----------|--------|----------|

|-------|-------------|-------------|-------------|------------------|----------------------|
| **26B MoE Q4_K_M** | GGUF 4-bit | ~14GB | 16GB | 40-60 tok/s | 25-40 tok/s |

### Recommended Deployment Stack

```
Inference: llama.cpp (GGUF) or Ollama (packaging)
    → Most portable, runs CPU+GPU, excellent Apple Silicon support
    → GGUF with imatrix calibration for best quality-per-bit

Alternative: ExLlamaV2 (EXL2) for NVIDIA-only deployments
    → Variable bits-per-weight, best NVIDIA performance

Packaging: Ollama custom Modelfile
    → Single command deployment: `ollama run agentshield-sigma`
    → Bundles model weights + system prompt + sampling config

API: OpenAI-compatible server (llama.cpp --server or Ollama API)
    → Drop-in replacement for cloud APIs
    → Enables integration with existing security tooling
```

### Tiered Model Strategy

The Gemma 4 family enables a tiered approach:

1. **E2B (2.3B active)** — "Sigma Lint": Fast validation, syntax checking, basic suggestions. Runs on any laptop. Free tier.
2. **E4B (4.5B active)** — "Sigma Assist": Rule completion, template filling, guided generation. Runs on any modern Mac/PC. Standard tier.
3. **26B MoE (4B active)** — "Sigma Generate": Full rule generation from threat descriptions, reasoning about detection logic. Requires 16GB+ RAM. Pro tier.
4. **31B Dense** — "Sigma Reason": Complex multi-rule generation, attack chain analysis, cross-rule optimization. Requires 20GB+ RAM / dedicated GPU. Enterprise tier.

Each tier can be independently fine-tuned with the same pipeline, sharing the reward function.

## 7. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Gemma 4 base model too weak for Sigma generation | Low | High | Fallback to 31B dense; benchmark early in Phase 1 |
| GRPO doesn't meaningfully improve over SFT | Medium | Medium | Invest in reward function quality; try DPO as fallback |
| Hallucinated field names / detection logic | High | High | Reward function penalizes non-existent fields; constrained decoding |
| Quantization degrades rule quality | Medium | Medium | Test each quant level on SigmaEval; use Q6_K if Q4 insufficient |
| Competitors replicate with the same base model | Medium | Low | Moat is in data + reward function + flywheel, not the base model |
| Google releases Gemma 5, obsoleting our work | Low | Medium | Pipeline is model-agnostic; re-run on new base in ~1 week |

## 8. Recommendation

**Start with the Gemma 4 26B MoE (A4B) as the primary target.** It offers the best performance-per-active-parameter ratio in the family, runs locally on a Mac Studio or RTX workstation, and the Apache 2.0 license removes all commercial friction.

The SFT + GRPO pipeline is well-supported by existing tooling (TRL, Unsloth, OpenRLHF), and the total compute budget (~$1,000) makes this accessible for iterative experimentation.

**The real investment is in data curation and reward function engineering** — not compute or model architecture. Our existing 45 rules and pytest conformance suite give us a meaningful head start. The first milestone should be Phase 0 (data prep) and a proof-of-concept SFT on the E4B model to validate the approach before committing to the full GRPO pipeline.
